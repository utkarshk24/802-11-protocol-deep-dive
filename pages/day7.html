CRC-32 Fundamentals

What is CRC?

A CRC is a mathematical checksum. You feed the frame data through a polynomial division algorithm, and the remainder is the checksum. If any bit flips, the remainder changes.

802.11 uses CRC-32 with polynomial: 0x04C11DB7

This creates a 32-bit (4-byte) FCS that's appended to every frame. When the receiver processes the frame + FCS through the same algorithm, the result must be zero (for most polynomials).

Why CRC, Not MD5 or SHA?

  • Speed: CRC is hardware-accelerated. Can process at gigabit speeds.
  • Burst error detection: Detects long runs of bit flips (noise bursts).
  • Not for security: CRC is NOT cryptographic. It detects accidental corruption, not attacks.

Calculating CRC-32 in Python

python crc_validator.py 
import binascii

def crc32_802_11(data):
    """Calculate CRC-32 for 802.11 frame."""
    return binascii.crc32(data) & 0xffffffff

# Example: frame data without FCS
frame_data = b'\x80\x41\x00\x00'  # Frame control + flags
fcs = crc32_802_11(frame_data)
print(f"CRC-32: {fcs:08x}")

# To validate, append FCS and recalculate
frame_with_fcs = frame_data + fcs.to_bytes(4, 'little')
result = crc32_802_11(frame_with_fcs)
print(f"Validation result: {result:08x}")  # Should be 0 for valid frame

Detecting Corrupted Frames

A frame with a bad CRC means the radio receiver detected corruption and should drop it. Wireshark and Scapy can tell you if a frame is corrupted.

python crc_validator.py 
from scapy.all import rdpcap, Dot11

packets = rdpcap("capture.pcap")

corrupted = 0
valid = 0

for pkt in packets:
    if pkt.haslayer(Dot11):
        # Check FCS bit (if available in radiotap)
        if hasattr(pkt, 'fcs_bad'):
            if pkt.fcs_bad:
                corrupted += 1
            else:
                valid += 1

print(f"Valid frames: {valid}")
print(f"Corrupted frames: {corrupted}")
print(f"Error rate: {corrupted / (valid + corrupted) * 100:.2f}%")

Exercise: Prove the CRC Works

  • Task 1: Take a known frame's bytes, compute the FCS, append it, and verify that re-running CRC-32 over frame+FCS yields the standard residual. Then flip a single bit and confirm the check fails.
  • Task 2: In a real capture, count how many frames your card flagged as fcs_bad. Correlate the FCS error rate with RSSI — do weak-signal frames fail more?
  • Task 3: Explain in one sentence why CRC-32 detects accidental corruption but provides zero protection against a deliberate attacker (foreshadowing the MIC in the security phase).
Hint: CRC is linear and unkeyed — an attacker who changes the payload can simply recompute a valid FCS. That's why encryption uses a keyed MIC (CCMP/GCMP) on top, which we cover in Lessons 40–42.
