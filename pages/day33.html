The Trigger Frame

A Trigger frame is the AP's command that orchestrates uplink multi-user transmission. It's addressed to a set of clients and tells each one precisely how to transmit:

  • Per-user RU allocation: which Resource Unit each client uses.
  • MCS & coding: the rate each client must use.
  • Target RSSI / power control: how much power each client should transmit at so all signals arrive at the AP at similar strength.
  • Trigger type: Basic, Beamforming Report Poll, MU-BAR, MU-RTS, etc.
Power alignment is the secret sauce.
If a near client blasts and a far client whispers, the loud one drowns the quiet one at the AP receiver. The Trigger frame's target-RSSI field tells each client how hard to transmit so all uplink RUs arrive at comparable power — essential for the AP to decode them together.
⬆️

The Uplink MU Sequence

text Trigger-Based Uplink 
AP  -> ALL   Trigger frame
               A: RU1, MCS4, tgtRSSI -55
               B: RU2, MCS2, tgtRSSI -55
               C: RU3, MCS6, tgtRSSI -55

  ... SIFS ...

A,B,C -> AP   HE TB PPDU  (all transmit simultaneously on their RUs)

  ... SIFS ...

AP  -> ALL   Multi-STA Block Ack  (acknowledges A, B, C at once)

This turns what used to be three separate contention-and-transmit cycles into one coordinated exchange — the uplink counterpart to downlink OFDMA.

⚖️

MU-EDCA: Contention for the MU Era

Classic EDCA (from 802.11e) governs how clients contend for the channel using per-access-category backoff. In Wi-Fi 6, the AP wants clients to prefer waiting for a Trigger rather than contending on their own — otherwise scheduled uplink and random uplink collide.

MU-EDCA provides a second set of EDCA parameters that the AP pushes to clients after they've participated in trigger-based access. These parameters use larger contention windows, making random channel access less attractive and nudging clients to rely on the AP's schedule. It's a gentle traffic-shaping mechanism that keeps the scheduled and random worlds from fighting.
