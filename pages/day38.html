WPA3 SAE
Handshake Deep Dive

SAE (Simultaneous Authentication of Equals) is the cryptographic core of WPA3-Personal. Today we go beyond Lesson 11's overview into the Dragonfly mechanics: the Password Element, the commit/confirm scalars and elements, PMK derivation, and the anti-clogging defense.

What Problem SAE Solves

In WPA2-PSK, the PMK is PBKDF2(password, SSID) — a fixed value. Capture one 4-way handshake and you can brute-force the password offline, forever, at billions of guesses per second. SAE is a Password-Authenticated Key Exchange (PAKE): each session derives a fresh, random shared secret, and an eavesdropper learns nothing to attack offline.

SAE gives forward secrecy.
Even if the password later leaks, previously captured traffic can't be decrypted, because each session's key came from ephemeral random values — not from the password alone. WPA2-PSK has no such protection.
🧮

The Dragonfly Exchange

SAE runs over a finite cyclic group (typically an elliptic curve like NIST P-256, group 19). Both peers are equal — either can start. The steps:

text SAE / Dragonfly Steps 
0. Both derive the Password Element (PWE):
     a point on the curve deterministically derived from
     (password, MAC_A, MAC_B) via hash-to-curve.
     (WPA3 uses Hash-to-Element / SSWU to avoid the timing
      leaks of the old "hunting-and-pecking" method.)

1. COMMIT (each peer):
     pick random  rand, mask
     scalar  = (rand + mask) mod q
     element = inverse(mask * PWE)         # a curve point
     send (scalar, element)

2. Verify peer's scalar/element are in range / on curve.

3. Compute shared secret K:
     K = rand * (peer_scalar * PWE + peer_element)
     k = F(K)                              # x-coord -> key material

4. CONFIRM (each peer):
     confirm = HMAC(k, own_scalar || peer_scalar ||
                       own_element || peer_element)
     send confirm; verify peer's confirm.

5. Derive PMK from k and the exchanged values.
   -> PMK feeds the normal 4-way handshake (Lesson 41).
🛡️

Anti-Clogging & Dragonblood

Because computing the PWE is expensive, an attacker could flood an AP with commit frames to exhaust its CPU. SAE defends with an anti-clogging token: under load, the AP replies "resend your commit with this token" (status 15), forcing the attacker to prove they can receive at their claimed address before the AP does costly work.

The 2019 Dragonblood research found side-channel and downgrade weaknesses in early SAE implementations (notably the hunting-and-pecking PWE derivation, which leaked timing info about the password). The fixes — constant-time Hash-to-Element and stricter group negotiation — are why modern WPA3 stacks are hardened. It's a good reminder: the protocol can be sound while implementations leak.

🧠 Knowledge Check
What are SAE's two phases?

Commit, then Confirm.

What is the Password Element (PWE)?

A curve point derived from the password and both MACs via Hash-to-Element.

What did Dragonblood exploit?

Timing side-channels in the old hunting-and-pecking PWE derivation; fixed by Hash-to-Element.
