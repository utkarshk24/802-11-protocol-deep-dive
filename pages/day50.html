Smart Roaming:
802.11k & 802.11v

Lesson 13 showed how 802.11r makes a roam fast. But a client still has to decide when and where to roam — and left alone, clients are notoriously "sticky." 802.11k and 802.11v are the assistance mechanisms that turn dumb, blind roaming into an informed, network-guided decision.

The 'Why' — The Sticky-Client Problem

The 802.11 standard leaves the roaming decision entirely to the client. Most clients cling to their current AP until the signal is nearly unusable (say −80 dBm), even when a much better AP is right overhead — because roaming means scanning every channel, which interrupts traffic. The result: a client dragging a terrible link across a building, hurting itself and everyone sharing that AP's airtime.

The network knows more than the client.
The infrastructure sees every AP's load, every neighbor's channel, and the whole RF topology. 802.11k lets it share that knowledge; 802.11v lets it nudge the client toward a better AP. Together with 802.11r's fast handoff, they're the "k/v/r" trio behind seamless enterprise roaming.
🗺️

802.11k — Radio Resource Measurement

802.11k adds measurement exchanges (via the Radio Measurement action frames from Lesson 15). The star feature is the Neighbor Report: instead of scanning all channels, the client asks the AP "who are my roaming candidates?" and gets back a curated list of nearby APs with their BSSIDs, channels, and capabilities.

text Neighbor Report Exchange 
STA -> AP   Neighbor Report Request
AP  -> STA  Neighbor Report Response:
              BSSID aa:bb:cc:00:11:23  channel 44  (same ESS)
              BSSID aa:bb:cc:00:11:24  channel 149 (same ESS)
              ... each with a BSSID Information field
                  (reachability, security match, mobility domain, ...)

Result: the client scans ONLY those 2 channels, not all ~25.
Scan time drops from hundreds of ms to tens -> roams sooner, cleaner.

Other 802.11k measurements include the Beacon Report (the client reports what APs it hears), Channel Load, and Noise Histogram — feeding the controller's view of the RF environment.

🧭

802.11v — BSS Transition Management

Where 802.11k informs, 802.11v steers. Its key tool is BSS Transition Management (BTM): the AP can send a client a BTM Request suggesting it move to a specific better AP, optionally with a "disassociation imminent" flag and a candidate list (often the same neighbor list from 802.11k).

text BTM Steering 
STA -> AP   BTM Query        "where should I go?" (optional)
AP  -> STA  BTM Request      "please move to BSSID X on ch 44"
                             flags: disassoc-imminent, candidate list,
                             preferred-candidate, validity timer
STA -> AP   BTM Response     accept (status 0) or reject (with reason)

If the client accepts, it roams (ideally via 802.11r for speed).
If it ignores the request, the AP MAY disassociate to force a re-roam.
BTM is a request, not a command.
A client can decline a BTM request — the standard can't force a roam. Well-behaved clients comply; poorly-behaved ones ignore it, which is why aggressive deployments pair BTM with a disassociation as a last resort. This is the polite, standards-based alternative to the old hack of deauthing sticky clients.
🐍

Compute It: Score Roam Candidates

Given a neighbor list and measured RSSIs, a simple roam-decision helper — the kind of logic a client's driver runs, and that you can replicate to audit roaming behavior:

python roam_decision.py 
def should_roam(current_rssi, candidates, hysteresis=8, floor=-75):
    """candidates: list of (bssid, rssi). Roam only if a candidate is
    meaningfully better AND current link is getting weak."""
    if current_rssi > floor:
        return None                       # link is fine, stay put
    best = max(candidates, key=lambda c: c[1], default=None)
    if best and best[1] >= current_rssi + hysteresis:
        return best                       # roam to clearly-better AP
    return None

current = -78
neighbors = [("aa:bb:cc:00:11:23", -60),   # much stronger
             ("aa:bb:cc:00:11:24", -74)]
target = should_roam(current, neighbors)
print("roam ->", target if target else "stay")

The hysteresis margin prevents "ping-pong" roaming between two similar APs — a candidate must be clearly better, not marginally, before the client commits.

Exercise: Audit Your Roaming

  • Task 1: Capture while walking between two APs on the same SSID. Do you see Neighbor Report or BTM action frames? Is your network actually using k/v?
  • Task 2: Measure the RSSI at which your device finally roams. Is it "sticky" (roaming only below −80 dBm)? Tune the floor/hysteresis in the code to match its behavior.
  • Task 3: Combine this with Lesson 13: trace a full roam and confirm the order — BTM suggestion → scan (narrowed by 11k) → 11r fast transition → reassociation.
