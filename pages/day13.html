Reassociation vs Association

A Reassociation Request (Subtype 2) is nearly identical to an association request but adds one crucial field: the Current AP Address (the BSSID the station is roaming from). This lets the new AP coordinate with the old AP to move buffered frames and tear down the old association.

The client decides when to roam — not the AP.
802.11 has no standard "you must leave" command (802.11v BSS Transition Management only suggests). Each client's driver uses its own logic: RSSI thresholds, retry rates, and load. Aggressive roaming means flapping; lazy roaming means "sticky client" problems.
🐢

Non-FT Roaming: The Slow Path

Without Fast Transition, roaming to a WPA2-Enterprise AP means repeating the entire join, including a full 802.1X/EAP exchange with the RADIUS server:

text Non-FT Roam (WPA2-Enterprise) 
Probe        ~10 ms
Open Auth    ~2  ms
Reassoc      ~2  ms
802.1X/EAP   200-700 ms   <-- full RADIUS round trips
4-way HS     ~20 ms
--------------------------
TOTAL        250-750 ms   <-- calls drop, video stutters

Anything over ~150 ms is audible on a VoIP call. That gap is exactly what 802.11r fixes.

⚡

802.11r Fast BSS Transition

Fast Transition (FT) pre-distributes key material across the mobility domain so the expensive 802.1X exchange happens once, at first connect. On every subsequent roam, keys are derived from a cached PMK-R0/PMK-R1 hierarchy — no RADIUS round trip.

  • Over-the-air FT: The station does the FT 4-way handshake directly with the target AP inside the Auth (alg=2) and Reassociation frames.
  • Over-the-DS FT: The station talks to the target AP through its current AP over the distribution system, then reassociates.
  • MDIE & FTIE: The Mobility Domain IE and Fast Transition IE carry the key context and nonces that make this possible.
text FT Roam 
Auth (alg=2, FT)   carries MDIE + FTIE + nonce
Reassoc Req        carries FTIE with MIC
Reassoc Resp       keys installed
--------------------------
TOTAL              < 50 ms   <-- seamless voice roaming
