The PLCP (Physical Layer Convergence Procedure) header is how the receiver synchronizes and knows what modulation to expect. Today we decode that header bit-by-bit.

The 'Why' — How Receivers Know What to Expect

The Synchronization Problem

A receiver is listening on a channel. Suddenly, a radio signal appears. The receiver has NO idea what's coming: is it a beacon? A data frame? What modulation? What length? What channel width?

The transmitter solves this by sending a PLCP preamble first: "Here comes data, MCS 5, length 1500 bytes." The receiver decodes the preamble, adjusts its demodulator, and then reads the data.

The PLCP header is a contract between transmitter and receiver.
"I'm sending you this data at this speed, with this length. Be ready." Without it, chaos.

802.11a/g PPDU (OFDM) Structure

Full Frame Anatomy

Every 802.11a/g/n data frame (PPDU) consists of: 

┌─────────────────────────────────────────────────────────────┐
│ Preamble + SIGNAL Header (sent at 6 Mbps = BPSK)          │
│                                                             │
│ ┌──────────────────┬──────────────────┬────────────────┐   │
│ │   L-STF          │     L-LTF        │   L-SIG        │   │
│ │ (4 µs)           │   (4 µs)         │  (4 µs)        │   │
│ │ Short Training   │  Long Training   │  SIGNAL Field  │   │
│ │ Field            │  Field           │                │   │
│ └──────────────────┴──────────────────┴────────────────┘   │
│                                                             │
│  Total preamble: 12 µs (always at 6 Mbps)                  │
└─────────────────────────────────────────────────────────────┘
┌─────────────────────────────────────────────────────────────┐
│ MAC Frame Payload (sent at variable MCS rate)              │
│                                                             │
│ ┌──────────────────────────────────────────────────────┐   │
│ │ MAC Header + Data + FCS (CRC)                        │   │
│ │ Can be 24–3840 bytes depending on MCS                │   │
│ └──────────────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────────────┘

L-STF: Legacy Short Training Field (4 µs)

The L-STF allows the receiver to detect the presence of a frame and estimate the channel. It consists of 10 repetitions of a short sequence, sent at 6 Mbps (BPSK).

  • Used for automatic gain control (AGC)
  • Gives the receiver time to adjust amplifier gain
  • Provides coarse timing synchronization

L-LTF: Legacy Long Training Field (4 µs)

The L-LTF refines synchronization and estimates the channel response (how multipath affects the signal). Consists of 2 OFDM symbols.

  • Fine timing and frequency offset correction
  • Channel estimation (enables equalization)
  • Allows 802.11n/ac devices to detect OFDM frame

L-SIG: Legacy SIGNAL Field (4 µs = 1 OFDM symbol)

The L-SIG tells the receiver: "What MCS are you about to see? How many bytes of data?" It's always sent at 6 Mbps (BPSK), so every receiver can decode it. 

L-SIG Field (24 bits):

Bit 0:      Rate (4 bits) = MCS index (0–7 for 802.11a/g)
Bit 4:      Reserved (1 bit)
Bit 5:      Length (12 bits) = number of octets in MAC frame
Bit 17:     Parity (1 bit) = even parity over bits 0–16
Bit 18:     Tail (6 bits) = 000000 (helps receiver sync)

Example: Rate = 0101 (MCS 5 = 48 Mbps), Length = 1500 bytes

The Tail Bits & Code Tail

The Tail field (6 zero bits after L-SIG) is part of the convolutional encoder's tail. It ensures the encoder returns to the zero state, allowing clean frame delineation.

Data Field: OFDM Symbols

How the Data is Sent

After the preamble and L-SIG, the transmitter sends the actual MAC frame data using the MCS specified in L-SIG. Each symbol is an OFDM symbol (4 µs for 802.11a/g/n). 

Example: 1500-byte frame at MCS 5 (48 Mbps)

L-SIG says: "48 Mbps, 1500 bytes"

Receiver calculates:
  Bits to send = 1500 bytes × 8 bits/byte = 12,000 bits
  MCS 5 = 48 Mbps = 48,000,000 bits/sec
  
  Coding overhead: With 3/4 coding, need 12,000 × 4/3 = 16,000 bits
  
  Bits per symbol = 4 (from 64-QAM in MCS 5)
  Coded bits per symbol = 4 × 52 subcarriers = 208 bits/symbol
  
  Symbols needed = 16,000 / 208 = 77 symbols
  
  Time = 77 symbols × 4 µs/symbol = 308 µs
  
  Total frame time = 12 µs (preamble+L-SIG) + 308 µs (data) = 320 µs

Pilot Subcarriers

Not all subcarriers carry data. 4 "pilot" subcarriers carry known reference signals that help the receiver track frequency offsets and phase errors during the frame.

  • Pilots are at subcarriers -21, -7, 7, 21
  • Receiver compares received pilots to expected values
  • Adjusts demodulation in real-time if frequency drifts

  Header revision: 0
  Header pad: 0
  Header length: 26
  Present flags: Flags, Rate, Channel, dBm Antenna Signal, Antenna
  Flags: 0x0010
  Data Rate: 48 Mbps              ← From L-SIG Rate field
  Channel Frequency: 5180 MHz     ← From regulatory domain
  Channel Flags: Turbo (a mode)
  SSI Signal: -45 dBm             ← Received power
  Antenna: 0

802.11 MAC Header
  Frame Control: 0x0841
  Duration/ID: 0x0000
  Receiver Address: aa:bb:cc:dd:ee:01
  Transmitter Address: aa:bb:cc:dd:ee:02
  Sequence number: 1234
  Fragment number: 0

Frame body: 1500 bytes

What Wireshark shows you:

  • Data Rate (48 Mbps): Decoded from L-SIG Rate field
  • RSSI (-45 dBm): Signal strength measured during preamble
  • Frame length: Decoded from L-SIG Length field (1500 bytes)
