Authentication Frames
(Open & SAE)

Before a station can associate, it must authenticate. In 802.11 this is a distinct frame exchange with its own algorithm numbers. Today we cover Open System auth, the deprecated Shared Key handshake, and the SAE (Dragonfly) exchange that powers WPA3.

The Authentication Frame

An Authentication frame is management Type 0, Subtype 11. The body begins with three fixed fields that drive the state machine:

  • Auth Algorithm Number: 0 = Open System, 1 = Shared Key, 2 = Fast BSS Transition, 3 = SAE.
  • Auth Transaction Sequence Number: which message in the exchange (1, 2, …).
  • Status Code: 0 = success; non-zero = rejected (e.g. 1 unspecified, 15 anti-clogging token needed).

Open System — Two Frames

Despite the name, Open System authentication provides no authentication at all. It's a formality: the client sends seq 1, the AP replies seq 2 with status 0. Real security (if any) happens later in the 4-way handshake. This is what WPA2-Personal and open networks use.

text Open System Exchange 
STA -> AP   Auth  alg=0  seq=1
AP  -> STA  Auth  alg=0  seq=2  status=0   (accepted)
              ... proceed to Association ...
🐉

SAE — Simultaneous Authentication of Equals

SAE (algorithm 3), based on the Dragonfly key exchange, is the heart of WPA3-Personal. It replaces WPA2's PSK-derived-from-password model with a password-authenticated key exchange that is resistant to offline dictionary attacks — capturing the handshake gives an attacker nothing to brute-force.

The Two-Phase Exchange

  • Commit (seq 1, both directions): Each side derives a Password Element (PWE) from the SSID + password, picks a random scalar/element, and sends a scalar and finite-field/EC element.
  • Confirm (seq 2, both directions): Each side proves it computed the same shared secret without revealing the password, sending a hash confirm value.
Why SAE beats WPA2-PSK.
In WPA2, capturing one 4-way handshake lets an attacker run an offline dictionary attack against the password forever. In SAE, each attempt requires a live interaction with the AP, and the anti-clogging token defends against flooding. Forward secrecy means a leaked password can't decrypt previously-captured traffic.
text SAE Exchange 
STA -> AP   Auth  alg=3  seq=1  (Commit: scalar, element)
AP  -> STA  Auth  alg=3  seq=1  (Commit: scalar, element)
STA -> AP   Auth  alg=3  seq=2  (Confirm: hash)
AP  -> STA  Auth  alg=3  seq=2  (Confirm: hash)  status=0
              ... shared PMK established, then 4-way handshake ...
🐍

Detecting Auth Type in a Capture

python auth_classifier.py 
from scapy.all import rdpcap, Dot11, Dot11Auth

ALG = {0: "Open System", 1: "Shared Key", 2: "FT", 3: "SAE"}

for pkt in rdpcap("capture.pcap"):
    if pkt.haslayer(Dot11Auth):
        a = pkt[Dot11Auth]
        print(f"{pkt[Dot11].addr2} -> {pkt[Dot11].addr1}  "
              f"alg={ALG.get(a.algo, a.algo):12s}  "
              f"seq={a.seqnum}  status={a.status}")

🧠 Knowledge Check
Auth algorithm numbers for Open System and SAE?

0 = Open System, 3 = SAE.

Why does capturing an SAE handshake not enable offline cracking?

SAE is a PAKE; each guess needs a live exchange, so there is no offline dictionary attack.

What security does Open System authentication provide?

None - it is a formality; real security comes later in the 4-way handshake.
