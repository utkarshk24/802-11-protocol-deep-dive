What the Scheduler Must Decide

Every transmission opportunity, the AP answers several coupled questions:

  • Which clients have queued traffic (DL) or want to send (UL, learned via buffer-status reports)?
  • What RU sizes to carve — many small RUs for many clients, or a few big ones?
  • Which client on which RU, considering each client's channel quality on that part of the band (frequency-selective fading means a client may be better on one RU than another).
  • What MCS per client, from their channel feedback.

The scheduler is proprietary — the standard defines the mechanism (RUs, triggers), vendors compete on the policy.

⚖️

Three Classic Fairness Models

text Scheduling Strategies 
Max Throughput   Always serve the client with the best channel.
                 -> Highest aggregate rate, but starves weak clients.

Equal Airtime    Give every client the same time on the medium.
                 -> Fair, but slow clients drag total throughput down.

Proportional     Weight by (achievable rate / recent throughput).
Fair             -> Balances the two: good clients get more, but
                    weak clients aren't starved. Common default.
The airtime-fairness insight.
Rate fairness (equal Mbps) forces the AP to spend disproportionate time on slow clients, which throttles everyone. Airtime fairness (equal time) is usually the healthier goal in mixed networks: fast clients finish quickly and free the medium.

A Toy Proportional-Fair Scheduler

python pf_scheduler.py 
class Client:
    def __init__(self, name, rate):
        self.name = name
        self.rate = rate          # achievable Mbps this slot
        self.avg  = 1.0           # smoothed recent throughput

def proportional_fair(clients, n_rus, alpha=0.1):
    """Assign n_rus RUs to the clients with highest rate/avg metric."""
    ranked = sorted(clients, key=lambda c: c.rate / c.avg, reverse=True)
    winners = ranked[:n_rus]

    for c in clients:
        served = c in winners
        got = c.rate if served else 0.0
        # exponential moving average of delivered throughput
        c.avg = (1 - alpha) * c.avg + alpha * got
    return [c.name for c in winners]

clients = [Client("fast", 600), Client("mid", 200), Client("slow", 30)]
for slot in range(5):
    print(f"slot {slot}: serve {proportional_fair(clients, n_rus=2)}")

Watch how the metric rate/avg lets the slow client occasionally win an RU even though its rate is low — its avg stays small, so its priority rises over time. That's proportional fairness in miniature.
