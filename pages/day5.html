The 'Why' — Theory → Practice

Reading Captures is a Real Skill

An engineer who can diagnose a network issue from a 10,000-frame PCAP is rare and valuable. Most engineers see a Wireshark capture and freeze: "How do I find the problem?"

With protocol knowledge + Wireshark skills, you instantly spot: "Device sent a deauth at frame 487, then probe request at 488. It roamed to a different AP." Or: "Auth frame at 120 succeeded, but assoc failed at 130. Check AP configuration."

PCAP analysis is where protocol knowledge becomes a superpower.
You read captures like a detective reads crime scenes. Every frame is evidence.

Wireshark Essentials for 802.11

Viewing the 802.11 PHY Layer

In Wireshark, right-click a frame → "View as Frame" → "Radiotap Header". You'll see:

  • Data Rate: Transmitted at X Mbps (from L-SIG)
  • Frequency: Which channel (2.4 GHz channel or 5 GHz)
  • dBm Antenna Signal: RSSI (received power)
  • Antenna: Which antenna received it
  • Flags: Whether frame was retransmitted (Retry flag)

Essential Wireshark Filters

FilterShows
wlan.fc.type == 0All Management frames
wlan.fc.type == 2All Data frames
wlan.fc.subtype == 0x08Beacons only
wlan.fc.subtype == 0x00Association Requests
wlan.fc.subtype == 0x01Association Responses
wlan.fc.subtype == 0x0bAuthentication frames
wlan.fc.subtype == 0x0cDeauthentication frames
wlan.da == aa:bb:cc:dd:ee:ffFrames destined for this MAC
wlan.ra == aa:bb:cc:dd:ee:ffFrames received by this MAC (RA)
wlan.fc.protected == 1Encrypted frames only
radiotap.dbm_antsignal < -80Weak signal frames (< -80 dBm)
wlan.fc.retry == 1Retransmitted frames

Step-by-Step Frame Analysis

1. Start with Management frames:

Filter: wlan.fc.type == 0

This shows every beacon, probe, auth, assoc, deauth frame.
Look for patterns: device probing → auth → assoc → connected.

2. Find the BSSID (AP MAC):

Look at beacon frames. The BSSID is the AP's MAC.
Beacons repeat every 100 ms (typical TIM interval).
If beacons stop, AP went down.

3. Trace a device connection:

Find the device MAC address.
Filter: wlan.da == device_mac OR wlan.ta == device_mac

Watch the sequence:
  1. Probe Request (looking for AP)
  2. Probe Response (AP replies)
  3. Auth Request
  4. Auth Response
  5. Assoc Request
  6. Assoc Response
  
If assoc fails, you'll see deauth. Find the reason code.
🖥

Real Example: Diagnosing a Roaming Issue

Scenario: A device keeps disconnecting and reconnecting.

Analysis:

Frame 120: Beacon from AP-01 (BSSID aa:bb:cc:01:02:03), RSSI -50 dBm
Frame 145: Data frames from device, RSSI -55 dBm
...
Frame 487: Disassociation from AP-01 (reason: "inactivity timeout")
Frame 488: Probe Request (device looking for new AP)
Frame 489: Beacon from AP-02 (BSSID aa:bb:cc:01:02:04), RSSI -45 dBm
Frame 500: Auth Request to AP-02
Frame 501: Auth Response OK
Frame 510: Assoc Request to AP-02
Frame 511: Assoc Response OK
Frame 520: Data frames from device (now connected to AP-02)

Diagnosis:
  - AP-01 disconnected device due to inactivity (no data for ~30 sec)
  - Device roamed to AP-02 (better signal: -45 dBm vs. -50 dBm)
  - Fix: Check AP-01 idle timeout setting (should be longer)

Without protocol knowledge: "Device is disconnecting" With protocol knowledge: "Device disconnected due to inactivity timeout on AP-01 at frame 487, roamed to AP-02"
