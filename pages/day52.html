The 802.1X Model: Three Players, One Port

802.1X is port-based network access control. Three roles:

  • Supplicant: the client device proving its identity.
  • Authenticator: the AP (or switch) — it's just a gatekeeper that relays messages and enforces the verdict.
  • Authentication Server: the RADIUS server that actually validates credentials against a directory (AD/LDAP) and decides yes/no.

The authenticator models the connection as two logical ports. The uncontrolled port passes only EAP authentication traffic; the controlled port (all real data) stays blocked until the RADIUS server says "authenticated." No valid credentials, no network — not even an IP address.

text 802.1X / EAP Flow 
Supplicant        Authenticator (AP)        RADIUS Server
    |  EAPOL-Start        |                        |
    |-------------------> |                        |
    |  EAP-Request/ID     |                        |
    | <------------------ |                        |
    |  EAP-Response/ID    |   RADIUS Access-Request |
    |-------------------> |----------------------> |
    |     ...EAP method exchange (tunneled)...      |
    |                     |   RADIUS Access-Accept  |
    |  EAP-Success        | <---------------------- |
    | <------------------ |  (+ derived MSK)        |
    |                     |                        |
    |==== 4-way handshake using PMK-from-MSK ====   | (Lesson 41)
    |  controlled port now OPEN -> data flows       |
EAP produces the key that seeds the 4-way handshake.
A successful EAP exchange derives a Master Session Key (MSK) known only to the supplicant and RADIUS server. The server ships it to the AP, the top 256 bits become the PMK, and the normal 4-way handshake (Lesson 41) takes over. So enterprise auth replaces the PSK/SAE source of the PMK — everything downstream is identical.

The EAP Method Zoo

EAP is a framework, not one protocol — the actual credential check is done by a chosen EAP method. The big ones:

MethodClient credentialServer cert?Notes
EAP-TLSclient certificateyesStrongest; mutual certs, no passwords. Needs PKI.
PEAP (MSCHAPv2)username/passwordyesPassword inside a TLS tunnel. Very common.
EAP-TTLSpassword / legacyyesLike PEAP; flexible inner methods.
EAP-FASTPAC / passwordoptionalCisco; tunnel via a protected access credential.
EAP-PWDpasswordnoDragonfly-based; no cert infrastructure.
Validate the server certificate — or it's all theater.
PEAP/TTLS protect the password inside a TLS tunnel, but only if the client verifies the RADIUS server's certificate. If a device is configured to "not validate," an attacker can stand up a fake AP + fake RADIUS, present any cert, and harvest the MSCHAPv2 exchange for offline cracking. EAP-TLS sidesteps this entirely — there's no password to steal.
🖥️

Where Enterprise Auth Meets Everything Else

  • WPA3-Enterprise 192-bit (Lesson 39): mandates EAP-TLS with P-384 certs and GCMP-256 — the no-passwords, uniform-192-bit configuration.
  • Fast roaming (Lessons 13 & 50): without 802.11r, every roam repeats the entire EAP+RADIUS exchange (hundreds of ms). FT caches the key hierarchy so roams skip RADIUS — the single biggest reason FT exists.
  • MFP (Lesson 42): WPA3-Enterprise requires management frame protection, so even enterprise networks resist the deauth attacks from Lesson 14.
python eap_type.py 
from scapy.all import rdpcap, EAP

EAP_TYPES = {1: "Identity", 4: "MD5", 13: "EAP-TLS", 21: "EAP-TTLS",
             25: "PEAP", 43: "EAP-FAST", 52: "EAP-PWD"}

def audit_eap(pcap):
    seen = {}
    for pkt in rdpcap(pcap):
        if pkt.haslayer(EAP):
            t = pkt[EAP].type
            seen[t] = seen.get(t, 0) + 1
    for t, n in sorted(seen.items()):
        print(f"{EAP_TYPES.get(t, f'type {t}'):12s}  {n} frames")
    # flag weak/legacy negotiation
    if 4 in seen:
        print("WARNING: EAP-MD5 seen -> weak, no mutual auth")

if __name__ == "__main__":
    import sys
    audit_eap(sys.argv[1])

🧠 Knowledge Check
What are the three 802.1X roles?

Supplicant (client), Authenticator (AP), and Authentication Server (RADIUS).

What does a successful EAP exchange produce for the handshake?

An MSK, from which the PMK is taken to seed the 4-way handshake.

Why must PEAP/TTLS clients validate the RADIUS certificate?

Otherwise a fake AP plus fake RADIUS can present any cert and harvest credentials.
